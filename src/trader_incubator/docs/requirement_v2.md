# 交易员孵化系统需求文档（增强版）

## 1. 文档信息
- 版本：v2.0
- 状态：Draft
- 基于文档：`requirement.md`
- 目标：将原始想法落地为可实现、可验证、可迭代的产品与工程需求

## 2. 项目目标
- 构建一个“交易员孵化与比赛”系统，在统一赛季规则下运行多名 AI 交易员。
- 支持交易时间内自动执行策略，非交易时间进行复盘与策略迭代。
- 提供可追溯的交易记录、收益统计与可视化面板。
- 系统由 AI Coding Harness 驱动，结合 Python 执行引擎和 CLI 工作流。

## 3. 范围定义
### 3.1 In Scope（本期范围）
- 赛季创建与配置管理。
- 交易员自动生成（通过 `帅帅` Skill）。
- 交易执行引擎（分钟级调度、信号执行、风控校验、落库）。
- 非交易时段任务（复盘、市场信息摘要、策略建议）。
- 收益率天梯与基础 Web 可视化。
- 审计日志、任务日志、错误告警（基础级）。

### 3.2 Out of Scope（本期不做）
- 实盘券商下单（本期默认仿真交易）。
- 高频毫秒级策略执行。
- 复杂衍生品（期权/期货）交易支持。
- 多租户权限体系（本期单组织内部使用）。

## 4. 核心角色与职责
### 4.1 世界系统（Orchestrator）
- 负责系统生命周期与任务编排。
- 调度交易阶段/非交易阶段工作流。
- 触发 `帅帅`、`交易所`、`交易员`技能或服务。

### 4.2 交易所（Execution Engine）
- Python 实现的策略执行框架。
- 每分钟触发一次执行循环。
- 拉取市场数据、请求交易员信号、风控校验、撮合（仿真）、记账。

### 4.3 帅帅（Trader Generator Skill）
- 根据赛季配置批量生成交易员。
- 为每位交易员生成“风格 + 初始策略 + 风控参数”。
- 确保任意两名交易员风格相似度不超过 50%。

### 4.4 交易员（Trader Agent）
- 具有名称、风格、策略、风控参数、持仓与资产状态。
- 交易时段：输出信号、执行策略、不允许在线改策略。
- 非交易时段：复盘、研究、提出策略更新草案（需版本化）。

### 4.5 可视化面板（Web UI）
- 展示赛季信息、交易员状态、收益天梯、交易明细、风控事件。

## 5. 关键业务对象与字段
### 5.1 赛季 Season
- `season_id`（唯一标识）
- `name`（赛季名）
- `market`（A_SHARE/HK/US）
- `start_time`（必填）
- `end_time`（可空；为空表示永续）
- `ticker_pool`（可空；为空表示市场全量）
- `status`（DRAFT/RUNNING/PAUSED/ENDED）

### 5.2 交易员 Trader
- `trader_id`
- `season_id`
- `name`
- `style_profile`（风格标签及向量）
- `strategy_version`
- `risk_profile`（仓位上限、单票上限、止损规则等）
- `state`（ACTIVE/SUSPENDED）

### 5.3 交易信号 Signal
- `signal_id`
- `trader_id`
- `timestamp`
- `ticker`
- `side`（BUY/SELL/HOLD）
- `confidence`（0~1）
- `reasoning`（摘要）

### 5.4 订单与成交（仿真）
- `order_id`、`signal_id`、`ticker`、`qty`、`price`
- `order_status`（CREATED/FILLED/REJECTED）
- `reject_reason`

### 5.5 账本与绩效
- `nav`（净值）
- `daily_return`、`cum_return`
- `max_drawdown`
- `sharpe`（可后续补充）

## 6. 功能需求
### FR-1 赛季管理
- 支持创建/查看/暂停/结束赛季。
- 校验必填字段：市场、开始时间。
- 若设置代码池，交易标的必须属于池内。

验收标准：
- 创建成功返回 `season_id`。
- 无效市场或时间配置时返回可读错误信息。

### FR-2 交易员生成
- 赛季启动时触发 `帅帅` 创建 N 名交易员。
- 生成结果含风格、初始策略、风控参数。
- 风格两两相似度必须 <= 50%。

验收标准：
- 生成失败有错误日志与重试机制。
- 相似度检查报告可追踪并持久化。

### FR-3 交易时段执行
- 分钟级调度循环：
  1) 拉取行情；
  2) 调用交易员生成信号；
  3) 风控校验；
  4) 仿真撮合；
  5) 更新持仓与账本；
  6) 写入日志。
- 交易时段内禁止策略变更。

验收标准：
- 每次循环有唯一 `run_id`。
- 任意一笔成交可追溯到 `signal_id` 与策略版本。

### FR-4 非交易时段任务
- 自动触发复盘流程：
  - 生成当日复盘摘要；
  - 汇总市场信息；
  - 产出策略调整建议（草案，不自动生效）。

验收标准：
- 每位交易员每天至少一份复盘记录。
- 策略草案需进入“待审核/待发布”状态。

### FR-5 收益天梯
- 按赛季实时展示交易员收益排名。
- 默认指标：累计收益率、回撤、胜率。

验收标准：
- 同一时刻排名规则一致且可重算。
- 指标刷新延迟不超过 1 分钟。

### FR-6 Web 可视化
- 页面至少包含：
  - 赛季概览页
  - 交易员列表/详情页
  - 交易记录页
  - 收益天梯页

验收标准：
- 页面可按赛季筛选数据。
- 交易记录支持按交易员、时间、标的筛选。

## 7. 非功能需求
- 可用性：核心调度成功率 >= 99%（日维度）。
- 性能：单赛季 20 名交易员下，分钟循环应在 30 秒内完成。
- 可观测性：结构化日志 + 关键指标（循环耗时、拒单率、错误率）。
- 可追溯性：订单、成交、收益可链路追踪。
- 安全性：密钥配置走环境变量，不入库不入日志。

## 8. 约束与规则
- 本期仅支持仿真交易，不直接连接券商。
- 策略更新必须版本化，不覆盖历史版本。
- 交易时间由市场日历决定，非交易时间才允许策略迭代。
- 未配置代码池时，默认市场全量标的可交易。

## 9. 异常处理
- 行情拉取失败：重试（指数退避），超限后跳过本轮并告警。
- 信号生成失败：记录错误并隔离单交易员，不阻断全局循环。
- 风控拒单：写明拒绝原因（超仓位/非代码池/价格异常等）。
- 数据写入失败：进入重试队列，保证最终一致性。

## 10. 里程碑建议
- M1：赛季与交易员生成闭环（可创建并启动赛季）
- M2：分钟级仿真交易闭环（可产生订单、成交、账本）
- M3：非交易复盘与策略草案流程
- M4：Web 面板与收益天梯
- M5：稳定性与观测性增强

## 11. 验收清单（整体验收）
- 能创建赛季并成功启动。
- 能生成多名差异化交易员（相似度合规）。
- 能在交易时段自动执行并形成可追溯交易记录。
- 能在非交易时段产出复盘与策略调整草案。
- 能在 Web 端查看赛季、交易员、交易记录与收益天梯。
